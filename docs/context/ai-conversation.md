Стек и почему именно он
Next.js 15 (App Router) — единственный разумный выбор для production: SSR для SEO страниц игры, Server Components для лидербордов без клиентского fetching, API Routes для проксирования ключей Claude и Stripe. Разворачивается на Vercel в один клик с правильными CORS-заголовками для Stockfish WASM.
chess.js — библиотека для валидации всех правил: рокировка, взятие на проходе, пат, мат. Весит ~50KB, работает в браузере. Это единственное, что нужно для игровой логики — ты не пишешь движок с нуля.

Визуальная часть доски — кастомный CSS-грид 8×8, а не Chessboard.js (слишком простой и некастомизируемый). Каждая клетка — <div>, фигуры — SVG-спрайты или PNG в готическом стиле. Движение фигур анимируется через GSAP: берём фигуру, вычисляем дельту позиции, делаем gsap.to(piece, { x, y, duration: 0.35, ease: "power2.out" }). Взятие — взрыв частиц через GSAP или простой CSS scale(0) + opacity(0) с эффектом огня/льда в зависимости от дома.

Rive — для живых 2D-персонажей. Дракон-тренер делается в Rive Editor как отдельный .riv файл: три состояния (idle, thinking, alert) переключаются из кода через riveInstance.stateMachineInput("state"). На Rive Community есть готовые драконы и рыцари — можно взять за основу. Важно: Rive используется для компаньона (дракон сбоку), а не для отрисовки каждой фигуры — это будет слишком тяжело и сложно синхронизировать с доской.
Stockfish 18 WASM (nmrugg/stockfish.js, версия от Chess.com) — запускается в браузере через Web Worker, не нагружает сервер вообще. Сложность задаётся через setoption name Skill Level value N (0-20). Запрос выглядит так: отправляешь FEN текущей позиции, получаешь лучший ход в UCI-нотации, конвертируешь в SAN через chess.js, применяешь на доске. Важный момент для Vercel: нужны заголовки Cross-Origin-Embedder-Policy: require-corp и Cross-Origin-Opener-Policy: same-origin — без них многопоточный WASM не заработает. Прописываются в next.config.js через headers().
Supabase закрывает сразу четыре задачи. Auth (Google OAuth + email) — это supabase.auth.signInWithOAuth. База данных — PostgreSQL с тремя ключевыми таблицами: users (id, username, elo_rating, city, house), games (id, white_id, black_id, pgn, result, duration, created_at), moves (game_id, ply, san, fen). Лидерборд — это SQL View с GROUP BY city. Realtime — канал на каждую игровую сессию: supabase.channel('game:' + gameId), broadcast для ходов, presence для статуса игроков. Edge Functions — для обновления Elo после окончания партии (классическая формула K-factor).
Claude API — AI-тренер-дракон. После игры фронтенд отправляет PGN в Next.js API Route, тот вызывает Anthropic API с системным промптом в стиле Вестероса. Во время игры дракон может давать подсказки (ограничено, чтобы не убивать опыт). Ответ стримится через Server-Sent Events и показывается как "речь дракона" в UI.

Мультиплеер — как это работает
Игрок A создаёт комнату → получает gameId → делится ссылкой /game/[gameId]. Игрок B переходит → оба подписываются на Supabase канал game:${gameId}. A делает ход: chess.js валидирует, анимация проигрывается локально, FEN и SAN отправляются через broadcast. B получает broadcast → chess.js применяет ход → анимация. Состояние игры хранится в Supabase DB как серия ходов — если страница обновилась, восстановление идёт через SELECT * FROM moves WHERE game_id = ? по PGN.
Авторитет хода — на клиенте (optimistic), но при получении хода от соперника всегда проверяем chess.js. Это стандартный подход, которого достаточно для chess: не FPS, задержка в 100-200ms никак не влияет на геймплей.

Насчёт Vercel — не обязательно
Vercel — это удобство, а не необходимость. Next.js прекрасно работает на VPS через PM2 + Nginx. Единственное, что нужно сделать руками — прописать CORS-заголовки для Stockfish WASM в nginx.conf:
nginxadd_header Cross-Origin-Embedder-Policy "require-corp";
add_header Cross-Origin-Opener-Policy "same-origin";
Без этих заголовков многопоточный WASM не запустится. С ними — всё работает. Cloudflare бесплатный тариф закроет тебе CDN для статики. Так что VPS на ps.kz + Cloudflare = полная замена Vercel.

Реально ли сделать качественную игру в браузере — главный вопрос
Честный ответ: да, и это намного проще, чем кажется. Но важно понять почему.
Lichess — второй по размеру шахматный сайт в мире — долгое время работал буквально на одном сервере за $183 в месяц. Миллионы партий. Это факт из их собственного доклада. Шахматы — это не Fortnite и не Counter-Strike. Это 64 клетки и максимум 32 фигуры. С точки зрения рендеринга это почти ничто.
Вот что реально используют chess.com и lichess для рендеринга доски — и это важно понять:
Доска — это обычный CSS-грид или flexbox из 64 div-ов. Не Canvas, не WebGL. Просто HTML. Фигуры — SVG или PNG файлы. Движение фигуры — это CSS transform: translate(Xpx, Ypx) с transition: 200ms ease, что браузер аппаратно ускоряет через GPU. GSAP делает то же самое, только с более точным контролем.
Lichess написан на Scala с асинхронным сервером, WebSocket-соединения обслуживает отдельный сервер через Redis, а клиент написан на TypeScript со snabbdom. Это их масштабируемая инфраструктура. Для твоего случая — хакатон + первые пользователи — ничего этого не нужно. GitHub
Для WebSocket в Node.js: в продакшн-среде с типичными нагрузками разница в производительности между ws и Socket.io обычно незначительна — оба комфортно справляются с нагрузкой на современном железе. Для шахмат это особенно верно: ход делается раз в несколько секунд, это буквально 20-50 байт JSON. Socket.io даст тебе комнаты, автореконнект и удобный API из коробки. Reintech

Что реально создаёт "quality experience" в браузерных шахматах
Исследовал chess.com — вот что там происходит под капотом с точки зрения UX, а не технологий:
Оптимистичные обновления. Когда ты делаешь ход, он показывается немедленно на твоей стороне, не дожидаясь подтверждения сервера. Если сервер отклонит — откатываешь. В шахматах сервер никогда не отклоняет валидный ход, поэтому это просто всегда работает.
Pre-move. Можешь кликнуть следующий ход, пока соперник думает — он выполнится автоматически после его хода. Это создаёт ощущение мгновенной реакции даже при задержке 200ms.
Звуки. Звук хода, взятия, шаха, мата. Это сильно влияет на ощущение "живой игры". Простые MP3-файлы через Web Audio API.
Подсветка. Последний ход подсвечивается. Когда берёшь фигуру — на допустимых клетках появляются точки. Это тоже просто CSS-классы на div-ах.
Плавность анимации. Не скорость, а именно плавность. ease-out на 200-300ms — фигура "приземляется" на место. Никаких teleport-переходов.
Всё это делается через обычный CSS и JavaScript. Никакого WebGL. Никакого Canvas. Canvas хорош для игр с тысячами объектов — Canvas-производительность остаётся константно низкой при росте числа объектов, тогда как SVG деградирует. Но у тебя 32 фигуры. SVG или PNG + CSS transition = идеально. Smus

Честно про твой стек: что делать
У тебя уже есть Next.js + PostgreSQL. Вот минимально необходимое что добавить:
Отдельный WebSocket-сервер — Node.js процесс с Socket.io, запущенный через PM2 рядом с Next.js. Nginx проксирует /socket.io/ на него, остальное — на Next.js. Это стандартная схема, хорошо документированная.
chess.js — npm-пакет для валидации правил. Ставишь, импортируешь, вызываешь game.move() — получаешь либо валидный ход, либо null. Вся логика шахмат за тебя.
Stockfish WASM — работает в браузере через Web Worker, на сервер нагрузки ноль. Берёшь stockfish.js (Chess.com версия, Stockfish 18), кладёшь в /public, загружаешь через new Worker('/stockfish-lite.js'). Отправляешь FEN, получаешь лучший ход.
PostgreSQL — у тебя уже есть. Добавляешь таблицы games, moves, users. Для мультиплеера состояние игры хранится в памяти сервера (объект Map<gameId, GameState>), в базу пишешь только завершённые партии и ходы.
Redis нужен только если WebSocket-серверов несколько. На одном VPS — не нужен.
FreedomPay — у вас уже есть модуль. Подключаете как есть через Next.js API route для Pro-подписки.

Единственное где буду честен в своей неуверенности
Сложная анимация персонажей в стиле GoT — Rive или GSAP-персонажи поверх каждой фигуры — это та часть, где я не уверен в соотношении затрат и результата для хакатона. Технически реализуемо, но Rive требует отдельной работы дизайнера-аниматора. Без готовых .riv-файлов ты будешь работать с SVG-спрайтами, которые анимируешь через CSS/GSAP. Это сработает и будет выглядеть хорошо — но не так, как Duolingo. Честная оценка: на хакатоне выигрышнее сделать идеально работающую игру с красивым GoT-визуалом через CSS/SVG, чем полусырой Rive с красивым дракончиком.
Дракона-AI-тренера (Rive персонаж сбоку) — вот это стоит сделать. Один Rive-файл для компаньона, который реагирует на события игры — это реалистично и эффектно.
